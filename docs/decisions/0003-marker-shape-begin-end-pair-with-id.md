# 0003 — Marker shape is a begin/end pair with explicit id

Both **Mentions** and **Annotations** are physically stored in markdown files as paired HTML comments wrapping the target region:

```
<!-- @sidebar {type} id="{id}" {key=value}*: {instruction} -->
target region
<!-- @sidebar end id="{id}" -->
```

`{type}` is one of `mention`, `note`, `suggestion`. `{id}` is generated by sidebar at insertion time (short hash, stable across content edits within the region). Key/value slots carry typed metadata (`verb="rephrase"`, `author="human"`, etc.). The instruction string is freeform after the colon.

We considered a single-marker form with an implicit region boundary (until next blank line, next heading, etc.) for terseness. We chose against it because the implicit boundary breaks the moment a region contains anything more complex than a single paragraph: lists, fenced code blocks, multi-paragraph quotes, and tables all break heuristic boundaries. Markdown tools that normalize whitespace would also silently invalidate single-marker regions. The begin/end pair is two lines of overhead instead of one; in the rendered editor the markers render in a side gutter so the visual cost is roughly zero, and the parser becomes trivial ("find begin, find matching end by id") rather than a years-long stream of edge-case fixes.

The explicit id is necessary because mentions and annotations are addressable by MCP tools (`mark_in_progress`, `resolve_mention`, `remove_annotation`). Sidebar generates the id on insertion; the agent never invents one. Ids are stable across content edits within the region, so file diffs do not invalidate outstanding work.
