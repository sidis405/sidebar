# 0007 — Stdio-first invite, with HTTP standalone preserved

Sidebar exposes its MCP server over two transports. The dominant path is stdio: the user's **Agent** spawns sidebar as a subprocess via an entry in a project-local `.mcp.json` written by `npx sidebar-md init`. The fallback path is HTTP: `npx sidebar-md` with no arguments runs standalone with no agent required (preserving Q2), and additional agents joining a multi-agent workspace attach to a running primary instance over HTTP.

We considered the original flow described in early drafts: sidebar starts first, prints an MCP config block, the user pastes it into their agent's MCP config, restarts the agent, and the agent reconnects. That is two file edits and one restart per project, plus a clipboard hop, every time the user wants to attach sidebar to a fresh project. Friction compounds across daily use. For a tool the author expects to use daily, that pattern is the thing that makes you stop using it.

We considered HTTP-only with `npx sidebar-md init` writing a project-local `.mcp.json`. This cuts the clipboard hop and the restart, but still requires the user to keep sidebar running in one terminal and start their agent in another. The dominant case (one agent, one project, frequent use) is two terminals instead of one.

We chose stdio-first because it collapses the dominant case to a single command. After one-time `init`, the user runs their agent in the project and sidebar comes up automatically as a subprocess. The browser opens. The agent is connected. No clipboard, no restart, no second terminal. The cost is internal: sidebar maintains two transports and a `.sidebar/connection.json` discovery file that routes a second `--stdio` invocation to proxy into the running HTTP primary rather than starting a duplicate instance. That complexity is contained inside sidebar and does not surface to users.

The standalone HTTP mode is not vestigial. It exists because [ADR-0001](0001-agent-runs-in-user-terminal-not-embedded.md) and Q2 state that a workspace is fully usable without any agent attached. Stdio-only would have made that impossible (no agent, no subprocess, no sidebar). Standalone HTTP is the entry point for "I want to read or edit project docs in sidebar without spinning up an agent" and is also the transport additional agents use when a primary instance is already running.

The project-local `.mcp.json` that `init` writes is committed to git by default. For a personal tool that may go open source, having `init` produce a file that cloning users pick up for free is the right ergonomic default. Users who want the file gitignored can move it to local-only by editing `.gitignore` after `init`; sidebar does not add it to `.gitignore` automatically.

The reason this is recorded as an **ADR** rather than left to the spec is reversibility. Once the author (or future users) build muscle memory around `cd project && claude`, switching the invite flow back to copy-paste-restart is a real regression that touches every project ever wired up. The decision binds the transport architecture, the boot ordering, and the user-facing command surface; all three are load-bearing for the V1 setup story. A future ADR may revisit if MCP-protocol developments (e.g. a standard agent-discovery mechanism for local servers) make a simpler design available.
